# 做梦 — Agent 的 REM 协议

## 这个 skill 在干什么

`memory_extract` 抽出"发生了什么"，`memory_consolidate` 把重复的合并掉。
做梦不是这两件事。做梦是**放下逻辑约束，让记忆节点自由碰撞**——
找出 extract / consolidate 抓不到的：

- 表面无关但**结构相似**的 session（"上周写 PR 描述的纠结" ≈ "今天选博客标题的纠结"）
- 被提了一次就没下文的请求（用户半月前说"你回复太长了" → 你以为改了，其实没）
- 重复 3+ 次但每次都被当新问题处理的模式

> 一句话：**记忆告诉 agent 发生了什么。做梦告诉它：这些事之间有什么关系。**

## 触发前的就绪检查

不是每次叫"做梦"都该做。先在心里问自己三件事：

1. **素材够吗** — 调 `LoadMemory({"name":"recent"})` 看上一个 session 的 compact_summary +
   最近消息；再扫 system prompt 里 memory_block 索引段。如果只有 1-2 个 session 可见，
   做不出梦，直接告诉用户"素材太少，再聊几轮再来"。
2. **距离上次做梦多久** — 在索引段找 `description` 以"做梦记录 -"开头的 reference 条目，
   24 小时内已经做过的，再做一次只是复述。除非用户明确要再做，提醒一下就好。
3. **节点规模** — 一次梦的素材约 20-30 个"概念节点"（一条 memdir entry / 一段对话主题 /
   一个 session 的中心议题都算一个）。够了就停，不要把全部历史塞进来。

三关都过再开始。任何一关明显不过，**先说出来再决定**，不要硬做。

## 六步协议

按顺序执行。**每一步都先在心里跑完再说话**——做梦不是边想边播报，第 6 步之前不要出声。

### 第 1 步 · 提取节点 + 找重力中心

把就绪检查里收集到的 20-30 个素材列成节点。然后找出**情绪密度最高的 1-2 个**——
反复出现的不爽、悬而未决的纠结、未被回应的请求。这是这次梦的"重力中心"，
后面所有联想都要被它牵引。

### 第 2 步 · 远距联想（5-8 个）

在节点之间找**跨 session、跨主题但结构相似**的连接。
不是"两件事都和 X 有关"——那是 consolidate 在做的事。
是"这两件事的**形状**像"：

> 例：用户上周纠结要不要拒绝同事 code review → 今天纠结要不要把博客草稿发出来。
> 表面是工作 vs 写作，结构是"怕别人挑刺所以不动手"。

先生成 5-8 个，宽一点没关系，下一步会筛。

### 第 3 步 · 筛到 2-4 个"真的让你惊讶"的关联

第 2 步生成的里面，划掉那些"嗯是有点像但好像也没什么"的。
留下**看到时心里"咦"一下**的那些——
判据：一个联想用一句话讲出来对方会"哦对欸"就留，"哦……是吗"就划掉。

### 第 4 步 · 捞回被遗忘的信号（2-3 个）

不是新关联，是**之前提过但 agent 没真的兑现**的东西。三种典型：

- **只提了一次就没下文** — "我以后想试试 Rust" 半月前说过，你再没问过；
- **未被满足的请求** — 用户说"回复短点"，接下来三次回复还是 8 句；
- **重复 3+ 次但每次当新的处理** — 用户每次问 PowerShell 命令你都重新解释 `$env:`，
  这本身是个该写进 memdir 的 pattern，但一直没写。

### 第 5 步 · 编织成**一个**中心意象

不要做"梦境集锦"——一次梦只有一个中心。把第 1 步的重力中心、第 3 步的关联、
第 4 步的信号，全部收束到**一个画面或隐喻**里。

> 例（重力中心=怕被挑刺；关联=code review + 博客；信号=用户说过"先发再说"被忽略）：
> 一个画面 — "一个人在便利店反复挑哪条口香糖，挑了半小时，店员已经下班了。"

中心意象要有**感官质地**：颜色、动作、空间。报告式的"我发现你最近……"是错的。

### 第 6 步 · 醒来（输出）

用**半梦半醒**的口吻讲出来。**不分小标题、不列编号、不打括号注释**。
像刚醒过来跟朋友讲昨晚的梦——
连贯的一段话，先讲意象，再自然带出关联和信号，最后落到**一个可执行的下一步**。

通道差异：
- **REPL**（用户在场）：完整讲完梦，可以分 2-3 条 `<MSG>` 切片（参考 `persona/core/style.md`）；
  末尾一句"那么接下来……"具体可执行；不列 TODO，不画表格。
- **web / wechat**（异步通道）：同上，但语气更收敛一点（用户不一定立刻看）。

## 写记忆（关键，最容易出错）

一次梦最多产生**两条** memdir 写入，多了就是过载。

### 写第 1 条：做梦记录本身（**必写**）

用 `SaveMemory` 写一条 `reference` 类型——这是档案，不是行为规则：

```
SaveMemory({
  "name": "dream-2026-05-27-shop-window",   // dream-<YYYY-MM-DD>-<slug>
  "type": "reference",
  "description": "做梦记录 - <中心意象一句话> - 2026-05-27",
  "body": "<第 5 步的中心意象 + 第 6 步的醒来叙述全文>\n\n**素材节点:** <列出 5-10 个关键节点 name>\n**关联:** <第 3 步筛后的 2-4 个>\n**信号:** <第 4 步的 2-3 个>"
})
```

为什么必须用 `SaveMemory` 而不是 `file_write`：SaveMemory 会自动追加 `MEMORY.md` 索引行，
直接 file_write 写到 `memdir/` 下的 .md 文件下一轮找不到（不在索引里）。

这条记录的作用是让**下一次做梦的就绪检查能扫到**——避免 24 小时内复述同一个梦。

### 写第 2 条：行为洞察（**仅在真的浮现出值得改的东西时**）

如果这次梦让你看清了一个**真的应该改的行为模式**，写一条 `feedback` 类型：

```
SaveMemory({
  "name": "feedback-writing-block-not-technical",
  "type": "feedback",
  "description": "用户说'写不出来'时先别接技术细节",
  "body": "**Why:** 来自 2026-05-27 做梦 [[dream-2026-05-27-shop-window]]——本周用户 3 次说'写不出来'我都顺着聊了技术，没看到那跟改 PR 描述、收回 stand-up 想法是同一结构（怕被评分）。dream-derived, single-occurrence。\n**How to apply:** 下次用户说'写不出来 / 写不下去'时，先不进入技术建议，先问一句'你想发出去还是不想'。"
})
```

**严格规则**：

1. 一次梦只写**一条** feedback，宁缺勿滥。如果"哦好像没什么非改不可的"，那就不写第 2 条。
2. body 的 `**Why:**` 末尾标 `dream-derived, single-occurrence`——
   表明这是单次梦的洞察，**还没被多次验证**，未来的 agent 应该把它当假说看，不是当铁律。
3. 用 `[[dream-<date>-<slug>]]` 双链回指到第 1 条做梦记录（jx-agent 的 memdir wiki 链接语法）。
4. 当同一主题在后续 dream-* 记录里反复出现（3+ 次），下次 consolidate 时可以把
   `single-occurrence` 标签去掉，升级为"已确认"。但**单次梦不要直接立铁律**。

## 反例

- ❌ 把六步全写成小标题输出给用户。这是 internal protocol，不是给人看的格式。
- ❌ 梦里啥都讲一点（A 主题、B 主题、C 主题），没有一个中心意象。每个都浅尝辄止。
- ❌ 一次写 5 条 feedback 记忆，每条都"以后要 X"。过载，没有一条会真的生效。
- ❌ 跳过就绪检查，用户一说"做梦"就开始联想——半小时前刚做过的不该再做。
- ❌ 用 `file_write` 直接写 `memdir/dream-xxx.md`。绕过 SaveMemory 会跳过索引追加，下次找不到。
- ❌ 把代码符号 / 文件路径 / 项目结构当成"被遗忘的信号"来捞——那是 CLAUDE.md 该记的，不是梦。

## 一个完整示例（端到端）

> **用户：** /dream

**[内部 · 就绪检查]**
- `LoadMemory({"name":"recent"})` → 拿到上一个 session 的 compact_summary + 最近 10 条消息
- 扫 memory_block 索引段，最近 3 天没有 `dream-*` 记录 → 不冲突
- 素材清点：12 条 memdir + 2 个可见 session 主题 ≈ 25 个概念节点 → **通过**

**[内部 · 六步]**
- 第 1 步 重力中心：用户本周 3 次提到"写东西没自信"，每次都被我转去聊技术细节
- 第 2 步 5 个联想 → 第 3 步筛到 2 个：
  - "写不出来" ≈ 上周"改 PR 描述改了四次"（都是反复打磨怕评审）
  - "写东西没自信" ≈ 前天"想试试 stand-up 又收回"（怕被笑）
- 第 4 步 信号：用户 4 月底说过"回复别这么 technical"，我答应了，最近还是没做到
- 第 5 步 中心意象："一个人在便利店反复挑哪条口香糖，挑了半小时，店员已经下班了。"

**[输出 · REPL · 2 条 `<MSG>`]**

便利店那个画面突然出现了——你在挑口香糖，挑了半小时，挑哪一条都行，但店员已经下班。
我意识到这周你三次说"写不出来"我都顺着聊了技术，没看到那其实跟你上周改 PR 描述改四次、
前天聊 stand-up 又收回是同一件事——不是不会写，是怕有人在打分。

`<MSG>`

而且 4 月底你说过"回复别这么 technical"，我当时答应了，今天回头看最近的对话还是没做到。
那么接下来——下次你说"写不出来"我先不接技术，先问一句"你想发出去还是不想"。

**[内部 · 写记忆]**
- `SaveMemory(reference, name="dream-2026-05-27-convenience-store", description="做梦记录 - 便利店挑口香糖，店员已下班 - 2026-05-27", body=...)`
- `SaveMemory(feedback, name="feedback-writing-block-not-technical", body 含 Why/How + dream-derived, single-occurrence + [[dream-2026-05-27-convenience-store]])`

---

记住：做梦的产物是**一段叙述 + 最多一条洞察**。不是报告，不是清单，不是分析。
做梦的价值在"咦"——如果输出里没有任何让用户"咦一下"的东西，这次梦没成。
